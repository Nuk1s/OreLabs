# OreLabs Review Checklist

## Purpose

This document defines the standard review process for all deliverables created within OreLabs Studio.

Every significant deliverable SHOULD be reviewed before being considered complete.

The objective of a review is to improve quality, maintain consistency and protect the project architecture.

---

# Review Philosophy

Reviews evaluate compliance with approved documentation.

Reviews SHALL focus on engineering quality rather than personal preference.

A review identifies issues.

A review does not redesign completed work.

---

# Review Categories

Every review SHOULD evaluate the following categories where applicable.

- Documentation
- Architecture
- Code
- Design
- Prompt Quality
- Repository Structure
- Standards Compliance

---

# General Review Checklist

Verify that:

✓ The objective has been completed.

✓ The output matches the specification.

✓ Approved documentation has been followed.

✓ No undocumented assumptions were introduced.

✓ Existing architecture remains consistent.

✓ Naming conventions are respected.

✓ Folder structure remains consistent.

✓ Dependencies are correctly documented.

✓ No unrelated changes were introduced.

✓ The deliverable is understandable by another AI.

---

# Documentation Review

Verify:

✓ Document follows OreLabs Standards.

✓ Metadata is complete.

✓ Internal references exist.

✓ Language is deterministic.

✓ Facts and ideas are separated.

✓ Revision History updated.

---

# Code Review

Verify:

✓ Code implements approved specification.

✓ Architecture preserved.

✓ No duplicated logic.

✓ No unnecessary complexity.

✓ Comments added where required.

✓ Naming conventions respected.

✓ No unrelated files modified.

✓ Implementation scope respected.

---

# Prompt Review

Verify:

✓ Role defined.

✓ Objective defined.

✓ Context sufficient.

✓ Requirements complete.

✓ Constraints explicit.

✓ Expected output defined.

✓ Success criteria measurable.

✓ Prompt follows Prompt Rules.

---

# Architecture Review

Verify:

✓ Responsibilities clearly separated.

✓ No circular dependencies.

✓ Scalability maintained.

✓ Folder organization preserved.

✓ Documentation reflects implementation.

---

# Repository Review

Verify:

✓ Files located correctly.

✓ Naming conventions respected.

✓ No orphan documents.

✓ No duplicate documents.

✓ References remain valid.

---

# AI Review

Verify:

✓ AI role respected.

✓ Workflow followed.

✓ AI Context considered.

✓ Required documentation read.

✓ Assumptions identified.

✓ Final response structured.

---

# Approval Levels

Level 1

Draft

Internal work only.

---

Level 2

Reviewed

Reviewed for quality.

---

Level 3

Approved

Accepted by the Founder.

---

Level 4

Locked

Changes require RFC or explicit Founder approval.

---

# Review Outcomes

A review SHALL produce one of the following outcomes.

Approved

Approved with Notes

Revision Required

Rejected

---

# Common Review Failures

Examples include:

- Missing documentation.
- Architecture violations.
- Incomplete requirements.
- Undefined assumptions.
- Ambiguous language.
- Incorrect repository location.
- Inconsistent naming.

---

# Review Responsibility

The reviewer SHALL evaluate quality.

The reviewer SHALL NOT silently modify approved architecture.

Recommendations SHALL be clearly distinguished from required corrections.

---

# Final Statement

Every review protects the long-term quality of OreLabs Studio.

Consistent reviews reduce technical debt, improve maintainability and ensure that future AI systems can understand and extend the project safely.
